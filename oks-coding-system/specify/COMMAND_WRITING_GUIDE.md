```yaml
agent: build
argument-hint: '<参数格式提示>'

```

| 项目          | 规范                  | 示例               |
| ------------- | --------------------- | ------------------ |
| 文件名        | `oks-{阶段}.md`       | `oks-vision.md`    |
| 命令名        | `/oks-{阶段}`         | `/oks-vision`      |
| description   | 简短描述（< 50 字符） | `创建项目愿景文档` |
| argument-hint | 参数格式提示          | `<项目名称>`       |

## 用途描述和使用范围

### 必须包含的内容

每个命令文件**必须**在开头包含：

1. **本命令用途**：一句话说明命令做什么
2. **使用范围**：
   - ✅ 适用场景（至少 2 个）
   - ❌ 不适用场景（至少 1 个）
3. **范围确认**：提示 AI 确认用户意图

### 标准模板

```markdown
# <命令标题>

**本命令用途**：<一句话说明命令做什么>

**使用范围**：

- ✅ 适用场景1
- ✅ 适用场景2
- ❌ 不适用于：不适用场景1（用 `/oks-xxx`）
- ❌ 不适用于：不适用场景2（用 `/oks-yyy`）

**用户输入**：`$ARGUMENTS`

在继续之前，你**必须**确认用户提供的信息与本命令的使用范围一致：

- 用户想<做什么>？
- <前置条件>是否满足？
- 如果用户想<不适用场景>，引导使用 `/oks-xxx`
```

### 示例

```markdown
# TDD 开发循环

**本命令用途**：使用 Red-Green-Refactor 循环开发领域模型（实体、值对象、业务规则）。

**使用范围**：

- ✅ 开发领域实体（Entity）
- ✅ 开发值对象（Value Object）
- ✅ 实现业务规则和状态机
- ❌ 不适用于：纯 CRUD 操作、数据传递对象（DTO）

**用户输入**：`$ARGUMENTS`

在继续之前，你**必须**确认用户提供的信息与本命令的使用范围一致：

- 用户想开发领域模型？
- 功能包含业务规则（而非纯数据传递）？
- 如果不适用，建议用户跳过 TDD 或使用其他命令
```

```markdown
## 分析用户意图

**用户输入**: $ARGUMENTS

在继续之前，你**必须**考虑用户输入：

### 意图识别

1. **<维度1>**：用户想<做什么>？
2. **<维度2>**：是否<明确某条件>？
3. **<维度3>**：<前置状态>是否已完成？

### 信息收集

如果用户输入不完整，询问以下信息：

| 优先级 | 问题      | 目的        |
| ------ | --------- | ----------- |
| 1      | <问题1>？ | 确定<目标1> |
| 2      | <问题2>？ | 确定<目标2> |
| 3      | <问题3>？ | 确定<目标3> |

### 收敛标准

满足以下条件后立即开始执行：

- [ ] 已确定<条件1>
```

## 交互式引导话术模板

### 必须包含的内容

每个命令文件**必须**在用户意图分析章节中包含：

1. **推荐提问顺序**：按优先级列出问题
2. **话术模板**：可直接使用的引导语言
3. **兜底策略**：当用户无法回答时的备选方案

### 标准模板

```markdown
### 交互式引导

当用户信息不完整时：

#### 推荐提问顺序

1. **范围确认**: 本次<动作>聚焦哪个模块？
2. **<维度2>确认**: 是否<条件>？
3. **<维度3>确认**: 有无<特殊要求>？

#### 可直接复用的话术模板

- 「请确认<信息1>。」
- 「<维度2>是否按<默认值>执行？如有差异请直接列出。」
- 「本次<动作><维度3>是什么？」

#### 兜底策略

| 选项 | 适用场景 | 推荐度 |
| ---- | -------- | ------ |
| A    | <场景A>  | 推荐   |
| B    | <场景B>  | 中     |
| C    | <场景C>  | 低     |
```

### 示例（来自 oks-design.md）

```markdown
### 交互式引导

当用户信息不完整时：

#### 0. 先读取技术基线

在追问用户前，先阅读 `docs/guides/tech-stack.md`，将其作为默认技术基线。

#### 1. 最小信息收集（必答）

- **项目/模块归属**：功能落在哪个项目？
- **技术栈来源**：用户指定或按默认执行
- **架构约定**：本次遵循的架构模式

#### 2. 推荐提问顺序

1. **范围确认**：本次设计聚焦哪个模块？
2. **技术确认**：是否沿用技术基线？
3. **架构确认**：是否沿用现有架构？

#### 3. 可直接复用的话术模板

- 「请确认该功能归属的项目/模块。」
- 「技术栈是否按默认执行？如有差异请直接列出。」
- 「本次设计沿用哪种架构约定？」

#### 4. 兜底策略

- 用户仅给功能名：先追问"模块归属 + 技术栈"
- 用户回答抽象：提供 2-3 个可选项

#### 5. 交互收敛标准

满足以下条件后立即停止追问：

- [ ] 已明确模块边界
- [ ] 已确定技术栈依据
- [ ] 已确定架构约定
```

## 参数验证和前置检查

### 参数验证

```markdown
## ⚠️ 参数验证

!`if [ -z "$ARGUMENTS" ]; then
  echo "❌ **错误: 缺少<参数名称>**"
  echo ""
  echo "**用法**: /oks-<命令> <参数>"
  echo ""
  echo "**示例**:"
  echo " /oks-<命令> <示例1>"
  echo " /oks-<命令> <示例2>"
  exit 1
fi`
```

### 前置条件检查

```markdown
## 🔒 前置条件检查

!`

# 使用统一的前置检查脚本

RESULT=$(bash oks-coding-system/scripts/check-prerequisites.sh --json --stage=<阶段> --feature="$ARGUMENTS" 2>&1)
if echo "$RESULT" | grep -q '"error"'; then
  MISSING=$(echo "$RESULT" | grep -o '"missing":\[[^]]*\]' | sed 's/"missing":\[/缺失: /; s/\]//; s/", "/\n  - /g' | sed 's/"//g')
  SUGGEST=$(echo "$RESULT" | grep -o '"suggestions":\[[^]]*\]' | sed 's/"suggestions":\[/建议: /; s/\]//; s/", "/\n → /g' | sed 's/"//g')
  echo ""
  echo "❌ **前置条件未满足**"
  echo ""
  echo "$MISSING"
echo ""
echo "$SUGGEST"
echo ""
echo "**解决方案**:"
echo " /oks-<前置命令> $ARGUMENTS"
exit 1
fi
`
```

### 前置条件矩阵

| 阶段           | 必需前置                 | 可选前置       |
| -------------- | ------------------------ | -------------- |
| vision         | 无                       | -              |
| user-story     | vision.md                | -              |
| design         | vision.md, user-story.md | -              |
| bdd            | user-story.md            | design.md      |
| tdd            | user-story.md            | design.md, bdd |
| implementation | tdd 完成                 | bdd            |
| e2e            | implementation 完成      | -              |
| optimization   | implementation 完成      | e2e            |

## Shell 脚本编写

### 变量命名

```bash
# ✅ 正确：使用下划线
VISION_COUNT=$(ls -1 "$VISION_DIR"/*-vision.md 2>/dev/null | wc -l)
PROJECT_NAME=$(basename "$file" -vision.md)

# ❌ 错误：使用星号（复制粘贴损坏）
VISION*COUNT=$(ls -1 "$VISION_DIR"/*-vision.md 2>/dev/null | wc -l)
PROJECT*NAME=$(basename "$file" -vision.md)
```

### 变量定义

```bash
# ✅ 正确：在使用前定义
REPO_ROOT=$(git rev-parse --show-toplevel 2>/dev/null || pwd)
VISION_DIR="$REPO_ROOT/docs/visions"

# ❌ 错误：使用未定义的变量
for file in "$VISION_DIR"/*-vision.md; do  # VISION_DIR 未定义
```

### 错误处理

```bash
# ✅ 正确：包含错误处理
REPO_ROOT=$(git rev-parse --show-toplevel 2>/dev/null || pwd)

# ✅ 正确：检查文件是否存在
if [ -f "$VISION_FILE" ]; then
  # 处理文件
else
  echo "⚠️ 文件不存在: $VISION_FILE"
fi

# ✅ 正确：静默错误输出
cat "$file" 2>/dev/null || echo "读取失败"
```

## 常见错误

### 1. 缺少用途描述和使用范围

```markdown
❌ 错误：

# BDD 场景设计

从用户故事创建可执行的 BDD 场景。

---

## 分析用户意图
```

```markdown
✅ 正确：

# BDD 场景设计

**本命令用途**：从用户故事创建可执行的 BDD 测试场景。

**使用范围**：

- ✅ 为已定义的功能需求创建 Gherkin 格式的测试场景
- ❌ 不适用于：编写代码实现

**用户输入**：`$ARGUMENTS`

在继续之前，你**必须**确认用户提供的信息与本命令的使用范围一致...

---

## 分析用户意图
```

### 2. 缺少范围确认提示

```markdown
❌ 错误：

在继续之前，你**必须**考虑用户输入...

✅ 正确：

在继续之前，你**必须**确认用户提供的信息与本命令的使用范围一致：

- 用户想创建 BDD 测试场景？
- 已有用户故事文档？
- 如果不是，引导用户使用正确的命令
```

### 3. 引用废弃命令

```markdown
❌ 错误：

运行 `/oks-workflow $ARGUMENTS` 执行完整工作流

✅ 正确：

依次执行 `/oks-user-story`, `/oks-design`, `/oks-tdd`, `/oks-implementation`
```

### 4. Shell 变量名损坏

```bash
❌ 错误（星号代替下划线）：

VISION*COUNT=$(ls -1 "$VISION_DIR"/*-vision.md 2>/dev/null | wc -l)

✅ 正确：

VISION_COUNT=$(ls -1 "$VISION_DIR"/*-vision.md 2>/dev/null | wc -l)
```

### 5. 路径引用不一致

```markdown
❌ 错误：

| 项目愿景 | [../../visions/{project}-vision.md](...) |

✅ 正确：

| 项目愿景 | [../vision.md](../vision.md) |
```
