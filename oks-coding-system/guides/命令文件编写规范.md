# OpenCode 命令文件编写规范

## 概述

OpenCode 命令文件允许你为重复性任务创建自定义命令。本文档详细阐述命令文件的编写规范，包括格式、编码、特殊符号使用等。

> 说明：本文档同时包含 **OpenCode 官方 Commands 语法** 与 **本项目约定扩展字段**。如需严格官方兼容，请以官方文档为准。

---

## 1. 文件格式与编码

### 1.1 文件格式

- **格式**：Markdown（`.md` 文件）
- **扩展名**：`.md`
- **文件命名**：
  - 命令名称由文件名决定（不含扩展名）
  - 支持点号分隔的命名空间（如 `speckit.plan.md` → `/speckit.plan`）
  - 示例：
    - `test.md` → `/test`
    - `speckit.plan.md` → `/speckit.plan`
    - `monitor-ci.md` → `/monitor-ci`

### 1.2 字符编码

- **编码格式**：UTF-8（支持中文、特殊字符等）
- **换行符**：Unix 风格（LF）

---

## 2. 文件结构

命令文件由两部分组成：

```
---
[frontmatter YAML 配置]
---

[正文内容（Markdown 格式的提示词模板）]
```

### 2.1 Frontmatter（前置配置）

Frontmatter 使用 YAML 格式，由 `---` 包围。

#### 官方字段（与 OpenCode Commands 一致）

| 字段 | 类型 | 说明 | 示例 |
| --- | --- | --- | --- |
| `description` | string | 命令描述，显示在 TUI 中 | `description: Run tests with coverage` |
| `agent` | string | 指定执行命令的代理 | `agent: build` |
| `model` | string | 覆盖默认模型 | `model: anthropic/claude-3-5-sonnet-20241022` |
| `subtask` | boolean | 强制作为子任务执行 | `subtask: true` |
| `template` | string | JSON 配置方式中定义命令模板（必需） | 见下方 JSON 示例 |

> 备注：在 Markdown 命令文件中，正文本身就是模板；在 JSON 配置中，`template` 是必需字段。

#### 项目扩展字段（非官方标准）

| 字段 | 类型 | 说明 | 示例 |
| --- | --- | --- | --- |
| `argument-hint` | string | 参数提示信息（由本项目命令系统消费） | `argument-hint: '[instructions] [--max-cycles N]'` |
| `handoffs` | array | 工作流交接配置（由本项目命令系统消费） | 见下文 |

#### Frontmatter 示例

**简单示例：**

```yaml
---
description: Run tests with coverage
agent: build
model: anthropic/claude-3-5-sonnet-20241022
---
```

**复杂示例（使用项目扩展字段）：**

```yaml
---
description: 执行实施规划工作流, 使用计划模板生成设计制品.
argument-hint: '[instructions] [--max-cycles N] [--timeout MINUTES]'
handoffs:
  - label: 创建任务
    agent: speckit.tasks
    prompt: 将计划分解为任务
    send: true
  - label: 创建检查清单
    agent: speckit.checklist
    prompt: 为需求创建质量检查清单
    send: true
---
```

### 2.2 正文内容

正文是发送给 LLM 的提示词模板，支持 Markdown 格式和特殊占位符。

**示例：**

```markdown
## 用户输入

$ARGUMENTS

## 任务

1. 运行测试套件
2. 分析失败原因
3. 建议修复方案
```

## 3. 特殊符号与占位符

### 3.1 参数占位符

| 占位符 | 说明 | 示例 |
| --- | --- | --- |
| `$ARGUMENTS` | 所有参数作为一个整体 | `/component Button` → `$ARGUMENTS` = `Button` |
| `$1` | 第一个参数 | `/create config.json src "{ }"` → `$1` = `config.json` |
| `$2` | 第二个参数 | 同上 → `$2` = `src` |
| `$3` | 第三个参数 | 同上 → `$3` = `"{ }"` |
| `$4`, `$5`... | 第 N 个参数 | 依此类推 |

**使用示例：**

```markdown
---
description: Create a new file with content
---

Create a file named $1 in the directory $2
with the following content: $3
```

**调用方式：**

```
/create-file config.json src "{ \"key\": \"value\" }"
```

### 3.2 Shell 命令输出

**语法：** `!`command\`\`

执行 bash 命令并将其输出插入到提示词中。

**示例 1：获取 git 状态**

```markdown
## 上下文

- **当前分支：** !`git branch --show-current`
- **当前提交：** !`git rev-parse --short HEAD`
```

**示例 2：运行测试并分析**

```markdown
---
description: Analyze test coverage
---

Here are the current test results:
!`npm test`

Based on these results, suggest improvements.
```

**注意事项：**

- 命令在项目根目录执行
- 输出会包含在发送给 LLM 的提示词中
- 如果命令失败，可能导致命令执行失败

### 3.3 文件引用

**语法：** `@filepath`

将文件内容直接包含在提示词中。

**示例：**

```markdown
---
description: Review component
---

Review the component in @src/components/Button.tsx.
Check for performance issues and suggest improvements.
```

**特性：**

- 支持相对路径和绝对路径
- 文件内容会被自动读取并包含在提示词中
- 适用于代码审查、分析等场景

## 4. 配置选项详解

### 4.1 Template（模板）

正文内容即为模板。也可以在 JSON 配置中通过 `template` 字段定义。

**JSON 配置方式（opencode.json）：**

```json
{
  "$schema": "https://opencode.ai/config.json",
  "command": {
    "test": {
      "template": "Run the full test suite with coverage report.\nFocus on failing tests and suggest fixes.",
      "description": "Run tests with coverage"
    }
  }
}
```

### 4.2 Description（描述）

显示在 TUI 命令列表中，帮助用户理解命令用途。

```yaml
---
description: Run tests with coverage and analyze failures
---
```

### 4.3 Agent（代理）

指定执行命令的代理类型。

```yaml
---
description: Review code changes
agent: plan
---
```

**常见代理类型：**

- `build`：构建相关任务
- `plan`：规划和设计任务
- 其他自定义代理

### 4.4 Subtask（子任务）

强制命令作为子任务执行，避免污染主上下文。

```yaml
---
description: Analyze codebase
subtask: true
---
```

**适用场景：**

- 临时性分析任务
- 不需要保留上下文的操作
- 独立的验证任务

### 4.5 Model（模型）

为特定命令覆盖默认模型。

```yaml
---
description: Complex analysis task
model: anthropic/claude-3-5-sonnet-20241022
---
```

## 5. 高级功能（项目扩展）

> 本章字段（`handoffs`、`argument-hint`）属于本项目扩展能力，不属于 OpenCode Commands 官方标准字段。

### 5.1 Handoffs（工作流交接）

用于复杂工作流，支持命令之间的交接。

```yaml
---
description: Execute planning workflow
handoffs:
  - label: 创建任务
    agent: speckit.tasks
    prompt: 将计划分解为任务
    send: true
  - label: 创建检查清单
    agent: speckit.checklist
    prompt: 为需求创建质量检查清单
    send: true
---
```

**字段说明：**

- `label`：交接项标签
- `agent`：目标代理
- `prompt`：交接时的提示词
- `send`：是否自动发送

### 5.2 Argument Hint（参数提示）

为用户提供参数使用提示。

```yaml
---
description: Monitor CI pipeline
argument-hint: '[instructions] [--max-cycles N] [--timeout MINUTES] [--branch BRANCH]'
---
```

### 6.2 带参数的命令

```markdown
- Use functional components with hooks
```

### 6.3 复杂工作流命令

**文件：** `.opencode/commands/analyze-project.md`

```markdown

agent: plan

---

## Project Analysis

### Context

- **Branch:** !`git branch --show-current`
- **Last Commit:** !`git log -1 --oneline`

### Tasks

1. Analyze package.json for dependencies
2. Review project structure
3. Identify potential improvements
4. Generate analysis report

### Files to Review

- @package.json
- @tsconfig.json
- @README.md
```

## 7. 命令文件存放位置

### 7.1 项目级命令

**路径：** `.opencode/commands/`

仅对当前项目有效。

### 7.2 全局命令

**路径：** `~/.config/opencode/commands/`

对所有项目有效。

### 7.3 优先级

项目级命令 &gt; 全局命令 &gt; 内置命令

**注意：** 自定义命令可以覆盖内置命令（如 `/init`, `/help` 等）。

## 8. 最佳实践

### 8.1 命名规范

- 使用小写字母和连字符：`run-tests.md`
- 支持命名空间：`speckit.plan.md`
- 名称应简洁且描述性强

### 8.2 描述规范

- 清晰说明命令用途
- 包含关键触发词（便于搜索）
- 示例：`description: Monitor CI pipeline and handle failures`

### 8.3 模板编写

- 使用 Markdown 结构化内容
- 合理使用标题（`##`, `###`）
- 使用列表和表格组织信息
- 保持提示词简洁明了

### 8.4 参数使用

- 提供清晰的 `argument-hint`
- 在模板中说明参数用途
- 处理参数为空的情况

### 8.5 Shell 命令

- 确保命令在项目根目录可执行
- 处理可能的错误情况
- 避免长时间运行的命令

### 8.6 文件引用

- 使用相对路径（更可移植）
- 确保文件存在
- 避免引用过大文件

### 9.3 常见问题

| 问题 | 原因 | 解决方案 |
| --- | --- | --- |
| 命令未找到 | 文件位置错误 | 检查文件是否在 `.opencode/commands/` |
| Shell 命令失败 | 命令不存在或无权限 | 检查命令语法和权限 |
| 文件引用失败 | 文件路径错误 | 使用正确的相对或绝对路径 |
| 参数未替换 | 占位符拼写错误 | 使用 `$ARGUMENTS`, `$1` 等正确格式 |

## 附录 B：Frontmatter 语法指南

> 以下附录内容整理自 `oks-coding-system/guides/Frontmatter 语法指南.md`，用于在本规范内一站式查阅。

Frontmatter 的语法可以拆解为三个层次来理解：最外层是**结构语法**（即分隔符），中间层是**数据类型语法**（即YAML的核心规则），最内层是**内容语法**（即你真正要写的各种字段）。

以下是一份详细的、综合性的 Frontmatter 语法指南。

### 1. 基础结构语法

Frontmatter 必须位于文件的最顶端，并以三个短划线 `---` 作为开始和结束的标记。

```yaml
---
# 这里写你的YAML数据
title: 我的文章标题
date: 2024-05-20
---
这是文章的正文内容，使用Markdown编写。
```

**关键规则**：

- **必须置顶**：开头的 `---` 必须是文件的**前三个字符**，前面不能有任何空格或空行。
- **独立成行**：开始和结束的 `---` 都必须单独占一行。

### 2. 核心数据类型语法

在 `---` 之间，你需要使用 YAML 语法来定义数据。YAML 是一门独立的标记语言，其语法非常直观，以下是 Frontmatter 中最常用的数据类型：

#### 字符串 (String)

大多数值默认就是字符串，可以直接书写。但如果字符串中包含**冒号** `:`**、引号、或特殊符号**，或者你想把数字（如版本号）、布尔值（如 `true/false`）强制作为字符串处理，**强烈建议用引号包裹**（单双引号均可，双引号更通用）。

```yaml
---
title: 这是一个普通标题
author: '张三'
version: '1.0' # 不加引号会被解析为数字
status: 'false' # 不加引号会被解析为布尔值 false
time: '14:30' # 不加引号可能导致解析错误
description: '包含: 特殊字符的句子'
---
```

#### 数字 (Number)

直接书写即可，可以是整数或小数。

```yaml
---
views: 12345
priority: 1
score: 98.5
---
```

#### 布尔值 (Boolean)

直接使用 `true` 或 `false`（不含引号）。

```yaml
---
published: true
draft: false
featured: false
---
```

#### 日期和时间 (Datetime)

YAML 本身支持 ISO 8601 格式的日期。具体使用哪种格式，也取决于你使用的工具（如静态生成器）的要求。常见格式有：

- 简单日期：`YYYY-MM-DD`
- 完整ISO格式：`YYYY-MM-DDTHH:mm:ss.sssZ`

```yaml
---
date: 2023-10-27
created: 2023-10-27T14:30:00.000Z
last_update: 2025-09-18 # Databricks 风格
---
```

#### 数组 (Array)

有两种等价的表示方式：

**行内写法**（适用于简单的列表）：

```yaml
---
tags: [JavaScript, HTML, CSS]
---
```

**列表写法**（更清晰，适用于较长的列表）：

```yaml
---
categories:
  - 技术
  - 前端
authors:
  - 张三
  - 李四
---
```

#### 对象 (Object)

用于将相关的属性分组，形成层级结构。

```yaml
---
author:
  name: 李四
  email: lisi@example.com
image:
  src: /images/cover.jpg
  alt: 封面图描述
  width: 1200
  height: 600
---
```

#### 多行文本 (Multiline)

YAML 提供两种方式处理长文本：

`|` **(保留换行)**：保留文本中的换行符。

`>` **(折叠换行)**：将多个连续的行合并为一个空格，直到遇到空行才真正换行。

```yaml
---
description: |
  这是第一行。
  这是第二行。
  这两行在最终渲染时会被放在不同的行里。

summary: >
  这是一个很长的句子，
  虽然在这里换行了，
  但最终会被合并成一句。
---
```

### 3. 常见/通用字段语义

虽然字段名完全由你或你使用的工具定义，但经过多年演化，以下字段已经成为约定俗成的"通用语义"：

| 字段名 | 语义类型 | 说明与示例 |
| --- | --- | --- |
| `title` | 字符串 | **页面/文章标题**。几乎所有工具都会读取它。`title: 我的博客` |
| `description` | 字符串 | **页面描述或摘要**。通常用于 SEO 或在列表中显示。`description: "这是一篇关于..."` |
| `date` | 日期 | **发布日期**。用于排序和显示。`date: 2024-05-20` |
| `tags` | 数组 | **标签**。用于内容分类。`tags: [Vue, React]` |
| `categories` | 数组/字符串 | **分类**。类似标签，但通常层级性更强。`categories: [技术, 前端]` |
| `draft` | 布尔值 | **是否为草稿**。如果为`true`，许多生成器不会发布它。`draft: true` |
| `layout` | 字符串 | **指定渲染模板**。静态生成器中常用。`layout: post` |
| `image` / `cover` | 字符串/对象 | **封面图或特色图片**。`image: /path/to/image.jpg` |
| `author` | 字符串/对象 | **作者信息**。`author: 张三` 或 `author: { name: 李四, email: ... }` |
| `head` | 数组 | **向页面HTML的**`<head>`**注入自定义标签**。常用于SEO。 |

### 4. 特定工具的专有语法

这是 Frontmatter 最强大也最需要留意的一点：**字段名和值的具体含义完全取决于你使用的工具**。

- **对于 VitePress 或 Rspress**，可以用 `layout: home` 和 `hero` 对象来构建一个极其复杂和美观的主页。

  ```yaml
  ---
  layout: home
  hero:
    name: '我的文档站'
    text: '一个神奇的网站'
    actions:
      - theme: brand
        text: 快速开始
        link: /guide/
  ---
  ```

- **对于 Bridgetown (Jekyll 衍生品)**，可以在 `_defaults.yml` 中定义全局默认值，而不必在每个文件里重复写相同的 `layout`。

- **对于 Databricks** 的技术文档，他们规定 `description` 字段是必需的，且有严格的长度限制。

### 5. 最佳实践与注意事项

1. **引用字符串**：当你对某个值的解析方式不确定时（例如包含`:`、`#`、`@`，或者你想让`1.0`、`false`保持为字符串），请**加上引号**。

2. **不要用 Tab**：YAML 只允许使用**空格**进行缩进。请确保你的编辑器将 Tab 键替换为空格。

3. **缩进敏感**：列表（`-`）和对象（`key: value`）的缩进必须对齐，否则会报错。

4. **注释**：你可以在 Frontmatter 中使用 `#` 进行注释，这些注释不会被解析为数据。

   ```yaml
   ---
   title: 我的文章 # 这是文章的标题
   # draft: true    # 这行被注释掉了，不会生效
   ---
   ```

### 总结

Frontmatter 的语法就是**由** `---` **包裹的、遵循 YAML 规范的数据块**。掌握了基础的数据类型（字符串、数字、数组、对象），再配合你所使用工具的手册，就可以熟练运用它了。
