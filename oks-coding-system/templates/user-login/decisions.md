## ADR-001：密码验证逻辑位置

### 元数据

- **状态**: ✅ 已采纳
- **决策日期**: 2026-03-11
- **决策者**: 开发团队
- **影响范围**: User 实体、UserService

### 背景

需要决定密码验证逻辑应该放在 User 实体还是 UserService 服务层。

**触发因素**:

- 实现 User.login() 方法时需要验证密码
- 需要决定业务逻辑的职责边界

**约束条件**:

- 密码已使用 bcrypt 加密存储
- 需要异步验证（bcrypt.compare 是异步的）
- 需要保持领域模型的纯粹性

### 备选方案

#### 方案 A: 放在 User 实体

**描述**:
在 User 实体中添加 `validatePassword(password: string): Promise<boolean>` 方法。

**优点**:

- ✅ 业务逻辑集中在实体中
- ✅ 实体包含完整的业务规则

**缺点**:

- ❌ 实体依赖外部库（bcrypt）
- ❌ 实体变得不够纯粹
- ❌ 违反依赖倒置原则

**成本**: 低（实现简单），但维护成本高

---

#### 方案 B: 放在 UserService 服务层（已选择）

**描述**:
在 UserService 中进行密码验证，User 实体只负责登录失败计数等业务规则。

**优点**:

- ✅ 实体保持纯粹，不依赖外部库
- ✅ 职责清晰：实体负责业务规则，服务层负责协调
- ✅ 符合 DDD 分层架构
- ✅ 易于测试（可以 mock bcrypt）

**缺点**:

- ❌ 密码验证逻辑不在实体中
- ❌ 需要在服务层调用实体的多个方法

**成本**: 中（需要更多协调），但维护成本低

---

### 决策

**选择方案**: 方案 B - 放在 UserService 服务层

**决策理由**:

1. 保持领域模型的纯粹性，实体不应该依赖基础设施层的库
2. 符合 DDD 分层架构原则
3. 更易于单元测试（可以 mock bcrypt）
4. 职责更清晰：实体负责业务规则（登录失败计数），服务层负责协调（密码验证）

**权衡考虑**:
虽然密码验证不在实体中，但这样做让实体更加纯粹，也更容易维护和测试。

### 影响

#### 正面影响

- ✅ 实体保持纯粹，易于理解和维护
- ✅ 服务层职责清晰
- ✅ 更易于测试

#### 负面影响

- ⚠️ 需要在服务层调用实体的多个方法
- ⚠️ 密码验证逻辑与实体分离

#### 风险

- 🚨 服务层逻辑可能变得复杂: 缓解措施是保持服务方法简洁，每个方法只做一件事

### 实施计划

1. **步骤 1**: User 实体只包含 loginAttempts、isLocked 等业务规则
   - 时间: 已完成
   - 负责人: 开发团队

2. **步骤 2**: UserService 实现 validatePassword() 方法
   - 时间: 进行中
   - 负责人: 开发团队

3. **步骤 3**: 编写测试验证实现
   - 时间: 待开始
   - 负责人: 开发团队

### 验证

**如何验证决策的正确性**:

- User 实体不依赖 bcrypt
- UserService 测试可以 mock bcrypt
- 代码职责清晰，易于理解

**成功标准**:

- User 实体测试覆盖率 > 80%
- UserService 测试覆盖率 > 80%
- 代码复杂度 < 10

## ADR-002：Token 存储方式

### 元数据

- **状态**: ✅ 已采纳
- **决策日期**: 2026-03-11
- **决策者**: 开发团队
- **影响范围**: 客户端、认证流程

### 背景

需要决定 JWT Token 应该存储在哪里。

### 备选方案

1. **方案 A: LocalStorage** — 简单易用，但存在 XSS 攻击风险
2. **方案 B: HttpOnly Cookie** — 更安全，防止 XSS，但需要 CSRF 防护
3. **方案 C: 内存存储** — 最安全，但刷新页面后需要重新登录

### 决策

**选择方案**: 方案 B - HttpOnly Cookie

**决策理由**:

1. 防止 XSS 攻击（JavaScript 无法访问 Cookie）
2. 自动发送到服务器（无需手动添加 Authorization 头）
3. 可以设置 Secure 和 SameSite 属性增强安全性

### 影响

- ✅ 更安全，防止 XSS 攻击
- ⚠️ 需要实现 CSRF 防护
- ⚠️ 跨域请求需要特殊配置

## ADR-003：登录失败计数存储位置

### 元数据

- **状态**: ✅ 已采纳
- **决策日期**: 2026-03-11
- **决策者**: 开发团队
- **影响范围**: 数据库、User 实体

### 背景

需要决定登录失败计数应该存储在数据库还是 Redis 缓存中。

### 备选方案

1. **方案 A: 数据库存储** — 持久化，简单，但性能较低
2. **方案 B: Redis 缓存** — 高性能，自动过期，但需要额外依赖

### 决策

**选择方案**: 方案 A - 数据库存储

**决策理由**:

1. 简单，不需要额外的 Redis 依赖
2. 持久化，重启后不丢失
3. 对于中小规模应用，数据库性能足够
4. 与用户数据在同一个事务中，保证一致性

### 影响

- ✅ 实现简单，易于维护
- ✅ 数据持久化，不丢失
- ⚠️ 性能不如 Redis（但可接受）
- 🚨 如果需要支持大规模用户，未来可以迁移到 Redis
