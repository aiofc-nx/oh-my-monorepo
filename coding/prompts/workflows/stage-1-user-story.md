# 阶段一：用户故事（User Story）

将功能需求转化为符合 INVEST 原则的用户故事。

---

## 当前任务

功能名称: **$ARGUMENTS**

## 项目上下文

当前分支: !`git branch --show-current`
最近提交: !`git log --oneline -3`

---

## 用户故事模板

```gherkin
作为 <角色>
我想要 <功能>
以便于 <价值>
```

---

## 执行步骤

### 1. 分析需求

理解 `$ARGUMENTS` 的业务价值：

- 这个功能解决什么问题？
- 目标用户是谁？
- 用户从中获得什么价值？

### 2. 识别角色

确定目标用户类型：

- 主要用户：直接使用功能的人
- 次要用户：间接受益的人
- 系统角色：其他系统或服务

### 3. 定义功能

明确用户想要做什么：

- 核心功能是什么？
- 需要哪些输入？
- 产生哪些输出？

### 4. 明确价值

说明用户从中获得的收益：

- 提高效率？
- 降低成本？
- 改善体验？
- 降低风险？

---

## INVEST 原则检查

| 原则            | 说明   | 检查点               | 通过标准          |
| --------------- | ------ | -------------------- | ----------------- |
| **I**ndependent | 独立性 | 故事之间没有依赖关系 | ✅ 可独立开发     |
| **N**egotiable  | 可协商 | 细节可以讨论         | ✅ 技术方案灵活   |
| **V**aluable    | 有价值 | 对用户有明确价值     | ✅ 用户价值明确   |
| **E**stimable   | 可估算 | 能够估算工作量       | ✅ 工作量 < 3 天  |
| **S**mall       | 足够小 | 一个迭代内能完成     | ✅ 单次提交可完成 |
| **T**estable    | 可测试 | 有明确的验收标准     | ✅ 验收标准明确   |

## 示例

**输入**: `用户登录`

**输出**: `docs/user-stories/user-login.md`

```markdown
# 用户故事: 用户登录

## 故事描述

作为注册用户
我想要使用邮箱和密码登录系统
以便于访问我的个人数据和功能

## 验收标准

- [ ] 用户可以使用有效邮箱和密码登录
- [ ] 登录失败时显示明确的错误信息
- [ ] 登录成功后跳转到首页
- [ ] 支持"记住我"功能（7 天免登录）
- [ ] 连续失败 5 次后锁定账户 15 分钟

## INVEST 验证

- [x] **I**ndependent - 独立于注册、登出功能
- [x] **N**egotiable - 技术方案灵活（JWT/Session）
- [x] **V**aluable - 核心功能，用户价值明确
- [x] **E**stimable - 工作量约 1 天
- [x] **S**mall - 单次提交可完成
- [x] **T**estable - 验收标准明确可测试

## 备注

- 认证方式：JWT Token
- 安全要求：密码加密存储
- 性能要求：登录响应 < 200ms
```

## 常见问题

### Q: 如果功能太大怎么办？

A: 拆分成多个小故事：

```
大故事: 用户管理
  ├─ 小故事 1: 用户注册
  ├─ 小故事 2: 用户登录
  ├─ 小故事 3: 用户资料编辑
  └─ 小故事 4: 用户登出
```

### Q: 如果验收标准不明确怎么办？

A: 使用 Given-When-Then 格式细化：

```
给定 [初始状态]
当 [执行操作]
那么 [期望结果]
```

### Q: 如果故事之间有依赖怎么办？

A: 重新组织或合并：

- 合并强依赖的故事
- 调整优先级顺序
- 识别并提取公共部分
