## 开发流程

### 开始新功能

```markdown
我要开发 {功能名称}。这是我的想法：[描述]。

请审查代码库并填写 `features/{feature}/design.md`，包括：

1. 技术设计（数据库、API、UI）
2. 用户故事
3. 边界情况
4. 成功标准
```

---

### 继续开发功能

```markdown
继续开发 {功能名称}。

请先阅读：

1. `features/{feature}/design.md` - 了解设计
2. `features/{feature}/implementation.md` - 了解当前进度

然后从上次中断的地方继续。
```

---

### 同步实现状态

```markdown
审查 `{feature}` 的已实现内容，并更新 `features/{feature}/implementation.md`：

1. 检查哪些代码已经实现
2. 更新"已完成"部分
3. 更新"进行中"和"下一步"
4. 更新测试状态
```

---

### 完成功能开发

```markdown
{功能名称} 开发完成。请：

1. 运行所有测试，确保通过
2. 更新 `features/{feature}/implementation.md` 状态为"已完成"
3. 检查是否需要更新 `design.md`
4. 生成功能文档
```

## 测试相关

### 生成单元测试

```markdown
为 `{component}` 编写单元测试，遵循：

1. 使用 AAA 模式 (Arrange-Act-Assert)
2. 覆盖所有公共方法
3. 包含正常、异常、边界情况
4. 测试命名清晰表达意图

参考 `features/{feature}/bdd-scenarios.md` 中的场景。
```

---

### 运行测试并修复

```markdown
运行测试并修复所有失败的测试：

1. 运行 `pnpm test` 查看失败测试
2. 分析失败原因
3. 修复问题（测试错误或实现错误）
4. 确保所有测试通过
```

---

### 检查测试覆盖率

```markdown
检查 `{feature}` 的测试覆盖率：

1. 运行 `pnpm test:coverage`
2. 识别未覆盖的代码
3. 补充测试以提高覆盖率到 80% 以上
4. 更新 `implementation.md` 中的覆盖率数据
```

---

### 运行 BDD 测试

```markdown
运行 `{feature}` 的 BDD 测试：

1. 运行 `pnpm test:e2e features/{feature}.feature`
2. 检查所有场景是否通过
3. 修复失败的场景
4. 更新 `implementation.md` 中的 BDD 测试状态
```

## 代码审查

### 代码质量审查

```markdown
从以下角度审查 `{feature}` 的改动：

1. **类型安全**: 是否有 any 类型？类型定义是否完整？
2. **错误处理**: 是否处理了所有可能的错误？
3. **安全性**: 是否有安全漏洞？（XSS、SQL 注入等）
4. **边界情况**: 是否处理了边界情况？
5. **代码复杂度**: 复杂度是否 < 10？
6. **代码重复**: 是否有重复代码？
```

---

### 性能审查

```markdown
审查 `{feature}` 的性能：

1. 是否有 N+1 查询问题？
2. 是否有不必要的重复计算？
3. 是否可以添加缓存？
4. 响应时间是否满足要求（< 200ms）？
5. 是否有内存泄漏风险？
```

---

### 安全审查

```markdown
审查 `{feature}` 的安全性：

1. **输入验证**: 所有用户输入是否验证？
2. **权限检查**: 是否有权限控制？
3. **敏感数据**: 敏感数据是否加密？
4. **SQL 注入**: 是否使用参数化查询？
5. **XSS**: 是否对输出进行编码？
6. **CSRF**: 是否有 CSRF 防护？
```

## 文档生成

### 生成内部文档

```markdown
为 `{feature}` 生成带截图的文档：

1. 在浏览器中打开该功能
2. 对关键 UI 状态截图
3. 将截图保存到 `features/{feature}/docs/screenshots/`
4. 创建/更新 `features/{feature}/docs/README.md`，包括：
   - 功能概览
   - 使用方式（带截图的分步说明）
   - 配置选项
   - 常见用例
   - 常见问题
```

---

### 更新 API 文档

```markdown
更新 `{feature}` 的 API 文档：

1. 列出所有新增/修改的 API
2. 为每个 API 编写文档：
   - 请求方法和路径
   - 请求参数
   - 响应格式
   - 错误码
   - 示例
3. 更新 API 文档文件
```

## 问题解决

### 调试失败测试

```markdown
测试 `{test-name}` 失败了。请：

1. 运行测试查看详细错误信息
2. 分析失败原因（测试错误 or 实现错误）
3. 修复问题
4. 确保测试通过
5. 更新 `implementation.md`
```

---

### 解决技术难题

```markdown
在实现 `{feature}` 时遇到问题：{问题描述}。

请：

1. 分析问题原因
2. 提出多个解决方案
3. 评估各方案的优缺点
4. 推荐最佳方案
5. 记录决策到 `decisions.md`
```

---

### 回滚决策

```markdown
需要回滚 ADR-XXX 的决策。请：

1. 分析回滚的影响
2. 制定回滚计划
3. 执行回滚
4. 更新 `decisions.md` 中 ADR-XXX 的状态为"已废弃"
5. 如有新的替代决策，创建新的 ADR
```

## 代码重构

### 重构代码

```markdown
重构 `{component}` 以提高代码质量：

1. 检查代码复杂度（目标 < 10）
2. 检查代码重复（目标 < 5%）
3. 拆分过长的方法（> 20 行）
4. 提取重复代码到公共方法
5. 改善命名
6. 确保所有测试仍然通过
```

---

### 消除技术债

```markdown
消除 `{feature}` 的技术债：

1. 查看 `features/{feature}/future-work.md` 中的技术债
2. 优先处理高优先级技术债
3. 每次处理一个技术债
4. 确保测试通过
5. 更新 `future-work.md`
```

## 特定场景提示词

### TDD 开发

```markdown
使用 TDD 方式实现 `{功能}`：

1. 🔴 Red: 先编写失败的测试
2. 🟢 Green: 用最简代码让测试通过
3. 🔵 Refactor: 优化代码，保持测试通过
4. 重复以上步骤
```

---

### 优化性能

```markdown
优化 `{feature}` 的性能：

1. 运行性能分析，识别瓶颈
2. 消除 N+1 查询
3. 添加缓存
4. 异步处理非关键路径
5. 验证性能提升（目标 > 20%）
```

## 使用建议

### 如何使用这些提示词

1. **复制粘贴**: 直接复制提示词到 AI 助手
2. **替换占位符**: 将 `{feature}`, `{component}` 等替换为实际值
3. **组合使用**: 可以组合多个提示词完成复杂任务
4. **自定义**: 根据项目需要调整提示词内容

### 创建新提示词

当发现常用的任务模式时，添加新提示词：

````markdown
### {任务名称}

\```markdown
{提示词内容}
\```
````
