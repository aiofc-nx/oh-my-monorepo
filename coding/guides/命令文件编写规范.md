# OpenCode 命令文件编写规范

## 概述

OpenCode 命令文件允许你为重复性任务创建自定义命令。本文档详细阐述命令文件的编写规范，包括格式、编码、特殊符号使用等。

---

## 1. 文件格式与编码

### 1.1 文件格式

- **格式**：Markdown（`.md` 文件）
- **扩展名**：`.md`
- **文件命名**：
  - 命令名称由文件名决定（不含扩展名）
  - 支持点号分隔的命名空间（如 `speckit.plan.md` → `/speckit.plan`）
  - 示例：
    - `test.md` → `/test`
    - `speckit.plan.md` → `/speckit.plan`
    - `monitor-ci.md` → `/monitor-ci`

### 1.2 字符编码

- **编码格式**：UTF-8（支持中文、特殊字符等）
- **换行符**：Unix 风格（LF）

---

## 2. 文件结构

命令文件由两部分组成：

```
---
[frontmatter YAML 配置]
---

[正文内容（Markdown 格式的提示词模板）]
```

### 2.1 Frontmatter（前置配置）

Frontmatter 使用 YAML 格式，由 `---` 包围。

#### 必需字段

| 字段          | 类型   | 说明                    | 示例                                   |
| ------------- | ------ | ----------------------- | -------------------------------------- |
| `description` | string | 命令描述，显示在 TUI 中 | `description: Run tests with coverage` |

#### 可选字段

| 字段            | 类型    | 说明                   | 示例                                               |
| --------------- | ------- | ---------------------- | -------------------------------------------------- |
| `argument-hint` | string  | 参数提示信息           | `argument-hint: '[instructions] [--max-cycles N]'` |
| `agent`         | string  | 指定执行命令的代理     | `agent: build`                                     |
| `model`         | string  | 覆盖默认模型           | `model: anthropic/claude-3-5-sonnet-20241022`      |
| `subtask`       | boolean | 强制作为子任务执行     | `subtask: true`                                    |
| `handoffs`      | array   | 交接配置（复杂工作流） | 见下文                                             |

#### Frontmatter 示例

**简单示例：**

```yaml
---
description: Run tests with coverage
agent: build
model: anthropic/claude-3-5-sonnet-20241022
---
```

**复杂示例（带参数提示和交接）：**

```yaml
---
description: 执行实施规划工作流, 使用计划模板生成设计制品.
argument-hint: '[instructions] [--max-cycles N] [--timeout MINUTES]'
handoffs:
  - label: 创建任务
    agent: speckit.tasks
    prompt: 将计划分解为任务
    send: true
  - label: 创建检查清单
    agent: speckit.checklist
    prompt: 为需求创建质量检查清单
    send: true
---
```

### 2.2 正文内容

正文是发送给 LLM 的提示词模板，支持 Markdown 格式和特殊占位符。

**示例：**

```markdown
## 用户输入

$ARGUMENTS

## 任务

1. 运行测试套件
2. 分析失败原因
3. 建议修复方案
```

## 3. 特殊符号与占位符

### 3.1 参数占位符

| 占位符        | 说明                 | 示例                                                   |
| ------------- | -------------------- | ------------------------------------------------------ |
| `$ARGUMENTS`  | 所有参数作为一个整体 | `/component Button` → `$ARGUMENTS` = `Button`          |
| `$1`          | 第一个参数           | `/create config.json src "{ }"` → `$1` = `config.json` |
| `$2`          | 第二个参数           | 同上 → `$2` = `src`                                    |
| `$3`          | 第三个参数           | 同上 → `$3` = `"{ }"`                                  |
| `$4`, `$5`... | 第 N 个参数          | 依此类推                                               |

**使用示例：**

```markdown
---
description: Create a new file with content
---

Create a file named $1 in the directory $2
with the following content: $3
```

**调用方式：**

```
/create-file config.json src "{ \"key\": \"value\" }"
```

### 3.2 Shell 命令输出

**语法：** `!`command``

执行 bash 命令并将其输出插入到提示词中。

**示例 1：获取 git 状态**

```markdown
## 上下文

- **当前分支：** !`git branch --show-current`
- **当前提交：** !`git rev-parse --short HEAD`
```

**示例 2：运行测试并分析**

```markdown
---
description: Analyze test coverage
---

Here are the current test results:
!`npm test`

Based on these results, suggest improvements.
```

**注意事项：**

- 命令在项目根目录执行
- 输出会包含在发送给 LLM 的提示词中
- 如果命令失败，可能导致命令执行失败

### 3.3 文件引用

**语法：** `@filepath`

将文件内容直接包含在提示词中。

**示例：**

```markdown
---
description: Review component
---

Review the component in @src/components/Button.tsx.
Check for performance issues and suggest improvements.
```

**特性：**

- 支持相对路径和绝对路径
- 文件内容会被自动读取并包含在提示词中
- 适用于代码审查、分析等场景

## 4. 配置选项详解

### 4.1 Template（模板）

正文内容即为模板。也可以在 JSON 配置中通过 `template` 字段定义。

**JSON 配置方式（opencode.json）：**

```json
{
  "$schema": "https://opencode.ai/config.json",
  "command": {
    "test": {
      "template": "Run the full test suite with coverage report.\nFocus on failing tests and suggest fixes.",
      "description": "Run tests with coverage"
    }
  }
}
```

### 4.2 Description（描述）

显示在 TUI 命令列表中，帮助用户理解命令用途。

```yaml
---
description: Run tests with coverage and analyze failures
---
```

### 4.3 Agent（代理）

指定执行命令的代理类型。

```yaml
---
description: Review code changes
agent: plan
---
```

**常见代理类型：**

- `build`：构建相关任务
- `plan`：规划和设计任务
- 其他自定义代理

### 4.4 Subtask（子任务）

强制命令作为子任务执行，避免污染主上下文。

```yaml
---
description: Analyze codebase
subtask: true
---
```

**适用场景：**

- 临时性分析任务
- 不需要保留上下文的操作
- 独立的验证任务

### 4.5 Model（模型）

为特定命令覆盖默认模型。

```yaml
---
description: Complex analysis task
model: anthropic/claude-3-5-sonnet-20241022
---
```

## 5. 高级功能

### 5.1 Handoffs（工作流交接）

用于复杂工作流，支持命令之间的交接。

```yaml
---
description: Execute planning workflow
handoffs:
  - label: 创建任务
    agent: speckit.tasks
    prompt: 将计划分解为任务
    send: true
  - label: 创建检查清单
    agent: speckit.checklist
    prompt: 为需求创建质量检查清单
    send: true
---
```

**字段说明：**

- `label`：交接项标签
- `agent`：目标代理
- `prompt`：交接时的提示词
- `send`：是否自动发送

### 5.2 Argument Hint（参数提示）

为用户提供参数使用提示。

```yaml
---
description: Monitor CI pipeline
argument-hint: '[instructions] [--max-cycles N] [--timeout MINUTES] [--branch BRANCH]'
---
```

## 6. 完整示例

### 6.1 简单命令

**文件：** `.opencode/commands/review.md`

```markdown
---
description: Review recent code changes
---

Review recent git commits:
!`git log --oneline -10`

Analyze these changes and suggest improvements.
```

### 6.2 带参数的命令

**文件：** `.opencode/commands/component.md`

```markdown
---
description: Create a new React component
---

Create a new React component named $ARGUMENTS with TypeScript support.

Requirements:

- Use functional components with hooks
- Include proper TypeScript types
- Add basic component structure
- Include a simple test file
```

**调用：** `/component Button`

### 6.3 复杂工作流命令

**文件：** `.opencode/commands/analyze-project.md`

```markdown
---
description: Analyze project structure and dependencies
agent: plan
argument-hint: '[--depth N] [--focus <area>]'
---

## Project Analysis

### Context

- **Branch:** !`git branch --show-current`
- **Last Commit:** !`git log -1 --oneline`
- **User Instructions:** $ARGUMENTS

### Tasks

1. Analyze package.json for dependencies
2. Review project structure
3. Identify potential improvements
4. Generate analysis report

### Files to Review

- @package.json
- @tsconfig.json
- @README.md
```

## 7. 命令文件存放位置

### 7.1 项目级命令

**路径：** `.opencode/commands/`

仅对当前项目有效。

### 7.2 全局命令

**路径：** `~/.config/opencode/commands/`

对所有项目有效。

### 7.3 优先级

项目级命令 > 全局命令 > 内置命令

**注意：** 自定义命令可以覆盖内置命令（如 `/init`, `/help` 等）。

## 8. 最佳实践

### 8.1 命名规范

- 使用小写字母和连字符：`run-tests.md`
- 支持命名空间：`speckit.plan.md`
- 名称应简洁且描述性强

### 8.2 描述规范

- 清晰说明命令用途
- 包含关键触发词（便于搜索）
- 示例：`description: Monitor CI pipeline and handle failures`

### 8.3 模板编写

- 使用 Markdown 结构化内容
- 合理使用标题（`##`, `###`）
- 使用列表和表格组织信息
- 保持提示词简洁明了

### 8.4 参数使用

- 提供清晰的 `argument-hint`
- 在模板中说明参数用途
- 处理参数为空的情况

### 8.5 Shell 命令

- 确保命令在项目根目录可执行
- 处理可能的错误情况
- 避免长时间运行的命令

### 8.6 文件引用

- 使用相对路径（更可移植）
- 确保文件存在
- 避免引用过大文件

## 9. 调试与测试

### 9.1 测试命令

在 TUI 中输入：

```
/your-command-name
```

### 9.2 带参数测试

```
/your-command arg1 arg2 arg3
```

### 9.3 常见问题

| 问题           | 原因               | 解决方案                             |
| -------------- | ------------------ | ------------------------------------ |
| 命令未找到     | 文件位置错误       | 检查文件是否在 `.opencode/commands/` |
| Shell 命令失败 | 命令不存在或无权限 | 检查命令语法和权限                   |
| 文件引用失败   | 文件路径错误       | 使用正确的相对或绝对路径             |
| 参数未替换     | 占位符拼写错误     | 使用 `$ARGUMENTS`, `$1` 等正确格式   |
