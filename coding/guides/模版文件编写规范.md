# OpenCode 模版文件编写规范

## 概述

本文档详细阐述 OpenCode 模版文件的编写规范。模版文件是可复用的 Markdown 文档，通过占位符系统与命令文件协同工作，用于自动生成各种开发文档和制品。

模版系统的核心价值：

- **标准化**：统一文档格式和结构
- **可复用**：一次定义，多次使用
- **自动化**：通过命令自动填充内容
- **可维护**：集中管理文档规范

---

## 1. 模版系统架构

### 1.1 基本概念

**模版文件**：包含占位符的 Markdown 文件，定义了文档的结构和格式。

**命令文件**：读取模版、填充占位符、生成最终文档的 OpenCode 命令。

**工作流程**：

```
用户输入 → 命令读取模版 → 填充占位符 → 生成文档
```

### 1.2 模版目录组织

模版文件可以组织在任意目录中，常见的组织方式：

**项目级模版**：

```
templates/                    # 项目根目录下的模版目录
├── spec-template.md         # 规范文档模版
├── plan-template.md         # 计划文档模版
├── tasks-template.md        # 任务列表模版
└── ...
```

**集中式模版**：

```
config/
└── templates/
    ├── docs/                # 文档类模版
    │   ├── spec-template.md
    │   └── plan-template.md
    ├── checklists/          # 检查清单模版
    │   └── review-template.md
    └── reports/             # 报告类模版
        └── summary-template.md
```

### 1.3 模版与命令的协作

命令文件通过读取模版并填充占位符来生成文档：

| 模版类型     | 典型用途         | 生成内容示例           |
| ------------ | ---------------- | ---------------------- |
| 规范模版     | 定义需求文档结构 | 功能规范、技术规范     |
| 计划模版     | 定义实施计划结构 | 开发计划、迭代计划     |
| 任务模版     | 定义任务分解结构 | 任务列表、待办事项     |
| 检查清单模版 | 定义质量检查项   | 代码审查清单、测试清单 |
| 报告模版     | 定义报告格式     | 状态报告、总结报告     |

## 2. 模版文件格式与编码

### 2.1 文件格式

- **格式**：Markdown（`.md` 文件）
- **扩展名**：`.md`
- **编码**：UTF-8（支持中文、特殊字符）
- **换行符**：Unix 风格（LF）

### 2.2 模版命名规范

- **格式**：`[类型]-template.md`
- **示例**：
  - `spec-template.md` - 功能规范模版
  - `plan-template.md` - 实施计划模版
  - `tasks-template.md` - 任务列表模版

## 3. 占位符系统

### 3.1 占位符格式

模版使用特定的占位符标记需要填充的内容。

#### 基本占位符

| 占位符               | 说明         | 示例                                   |
| -------------------- | ------------ | -------------------------------------- |
| `[FEATURE]`          | 功能名称     | `[FEATURE]` → `User Authentication`    |
| `[FEATURE NAME]`     | 完整功能名称 | `[FEATURE NAME]` → `用户认证系统`      |
| `[DATE]`             | 日期         | `[DATE]` → `2026-03-11`                |
| `[###-feature-name]` | 分支名称     | `[###-feature-name]` → `001-user-auth` |

#### 结构化占位符

| 占位符                      | 说明         | 使用场景       |
| --------------------------- | ------------ | -------------- |
| `[PROJECT_NAME]`            | 项目名称     | 章程文档       |
| `[PRINCIPLE_N_NAME]`        | 原则名称     | 章程文档       |
| `[PRINCIPLE_N_DESCRIPTION]` | 原则描述     | 章程文档       |
| `[SECTION_N_NAME]`          | 章节名称     | 章程文档       |
| `[SECTION_N_CONTENT]`       | 章节内容     | 章程文档       |
| `[CATEGORY_N]`              | 分类名称     | 检查清单       |
| `[CHK###]`                  | 检查项编号   | 检查清单       |
| `[T###]`                    | 任务编号     | 任务列表       |
| `[US#]`                     | 用户故事编号 | 任务列表、规范 |
| `[FR-###]`                  | 功能需求编号 | 功能规范       |
| `[SC-###]`                  | 成功标准编号 | 功能规范       |

### 3.2 条件占位符

#### NEEDS CLARIFICATION 标记

用于标记需要进一步澄清的内容。

**语法**：`[NEEDS CLARIFICATION: 描述]`

**示例**：

```markdown
- **FR-006**: 系统必须通过 [NEEDS CLARIFICATION: 未指定认证方法 - 电子邮件/密码, SSO, OAuth?] 认证用户
- **FR-007**: 系统必须保留用户数据 [NEEDS CLARIFICATION: 未指定保留期限]
```

**处理流程**：

1. 命令识别 `NEEDS CLARIFICATION` 标记
2. 生成澄清问题
3. 用户回答后填充内容
4. 移除标记

#### 可选区块标记

用于标记可选内容区块。

**示例**：

```markdown
<!--
  需要操作: 将此部分内容替换为项目的技术细节.
  此处的结构以咨询性质呈现, 用于指导迭代过程.
-->
```

## 4. 注释与指令

### 4.1 HTML 注释

模版使用 HTML 注释提供指导和说明。

#### 指导性注释

```markdown
<!--
  需要操作: 本节内容表示占位符.
  请用正确的功能需求填写.
-->
```

#### 重要说明注释

```markdown
<!--
  ============================================================================
  重要说明: 以下任务仅为说明目的的示例任务.

  命令必须根据以下内容替换为实际任务:
  - 来自 spec.md 的用户故事(及其优先级 P1, P2, P3...)
  - 来自 plan.md 的功能需求
  - 来自 data-model.md 的实体
  - 来自 contracts/ 的端点

  请勿在生成的 tasks.md 文件中保留这些示例任务.
  ============================================================================
-->
```

### 4.2 特殊标记

| 标记                    | 说明                     | 使用场景     |
| ----------------------- | ------------------------ | ------------ |
| `<!-- 需要操作: -->`    | 需要用户或命令填充的内容 | 所有模版     |
| `<!-- 重要说明: -->`    | 关键信息说明             | 所有模版     |
| `<!-- 注意: -->`        | 注意事项                 | 所有模版     |
| `<!-- 示例: -->`        | 示例说明                 | 章程模版     |
| `<!-- 手动添加开始 -->` | 手动内容开始标记         | 代理文件模版 |
| `<!-- 手动添加结束 -->` | 手动内容结束标记         | 代理文件模版 |

## 5. 模版结构规范

### 5.1 功能规范模版（spec-template.md）

#### 文档头部

```markdown
# 功能规范: [FEATURE NAME]

**功能分支**: `[###-feature-name]`
**创建时间**: [DATE]
**状态**: 草稿
**输入**: 用户描述: "$ARGUMENTS"
```

#### 必需章节

1. **用户场景与测试**
   - 用户故事（按优先级 P1, P2, P3 排序）
   - 验收场景（Given-When-Then 格式）
   - 边界情况

2. **需求**
   - 功能需求（FR-001, FR-002...）
   - 关键实体

3. **成功标准**
   - 可衡量的结果（SC-001, SC-002...）

#### 用户故事格式

```markdown
### 用户故事 1 - [简要标题] (优先级: P1)

[用通俗语言描述这个用户旅程]

**优先级原因**: [解释价值以及为什么具有此优先级]

**独立测试**: [描述如何独立测试]

**验收场景**:

1. **给定** [初始状态], **当** [操作时], **那么** [预期结果]
2. **给定** [初始状态], **当** [操作时], **那么** [预期结果]
```

### 5.2 实施计划模版（plan-template.md）

#### 文档头部

```markdown
# 实施计划: [FEATURE]

**分支**: `[###-feature-name]` | **日期**: [DATE] | **规范**: [link]
**输入**: 来自规范文档的功能需求

**注意**: 此模板由命令自动填充.
```

#### 必需章节

1. **摘要**
2. **技术背景**
   - 语言/版本
   - 主要依赖
   - 存储
   - 测试
   - 目标平台
   - 项目类型
   - 性能目标
   - 约束条件
   - 规模/范围
3. **章程检查**
4. **项目结构**
   - 文档结构
   - 源代码结构
5. **复杂度跟踪**（可选）

#### 技术背景示例

```markdown
**语言/版本**: [例如: Python 3.11, Swift 5.9, Rust 1.75 或 NEEDS CLARIFICATION]
**主要依赖**: [例如: FastAPI, UIKit, LLVM 或 NEEDS CLARIFICATION]
**存储**: [如适用, 例如: PostgreSQL, CoreData, 文件 或 N/A]
**测试**: [例如: pytest, XCTest, cargo test 或 NEEDS CLARIFICATION]
```

### 5.3 任务列表模版（tasks-template.md）

#### Frontmatter

```yaml
---
description: '功能实现任务列表模板'
---
```

#### 任务格式

**基本格式**：`- [ ] [ID] [P?] [Story] 描述`

| 标记     | 说明         | 示例                                                 |
| -------- | ------------ | ---------------------------------------------------- |
| `[ ]`    | 未完成任务   | `- [ ] T001 创建项目结构`                            |
| `[x]`    | 已完成任务   | `- [x] T001 创建项目结构`                            |
| `[P]`    | 可并行执行   | `- [ ] T003 [P] 配置代码检查`                        |
| `[US#]`  | 所属用户故事 | `- [ ] T010 [US1] 创建模型`                          |
| `🎯 MVP` | MVP 标记     | `## 阶段 3: 用户故事 1 - [Title] (优先级: P1)🎯 MVP` |
| `⚠️`     | 警告标记     | `### 用户故事 1 的测试(可选 - 仅在要求测试时)⚠️`     |

#### 阶段结构

```markdown
## 阶段 1: 设置(共享基础设施)

**目的**: 项目初始化和基本结构

- [ ] T001 根据实施计划创建项目结构
- [ ] T002 使用 [framework] 依赖项初始化 [language] 项目
- [ ] T003 [P] 配置代码检查和格式化工具

---

## 阶段 2: 基础(阻塞前置条件)

**目的**: 在任何用户故事可以实施之前必须完成的核心基础设施

**⚠️ 关键**: 在此阶段完成之前, 无法开始任何用户故事工作

- [ ] T004 设置数据库架构和迁移框架
- [ ] T005 [P] 实施身份验证/授权框架

**检查点**: 基础就绪 - 现在可以开始并行实施用户故事

---

## 阶段 3: 用户故事 1 - [Title](优先级: P1)🎯 MVP

**目标**: [此故事交付内容的简要描述]

**独立测试**: [如何验证此故事独立运行]
```


### 5.5 项目章程模版（constitution-template.md）

#### 核心原则格式

```markdown
### [PRINCIPLE_1_NAME]

<!-- 示例: I. 库优先 -->

[PRINCIPLE_1_DESCRIPTION]

<!-- 示例: 每个功能都作为独立的库开始; 库必须是自包含的, 可独立测试的, 有文档的 -->
```

#### 治理格式

```markdown
## 治理

<!-- 示例: 章程优先于所有其他实践; 修订需要文档, 批准, 迁移计划 -->

<!-- 示例: 所有 PR/审查必须验证合规性; 复杂性必须证明合理 -->

**版本**: [CONSTITUTION_VERSION] | **批准日期**: [RATIFICATION_DATE] | **最后修订**: [LAST_AMENDED_DATE]
```

### 5.6 代理文件模版（agent-file-template.md）

#### 手动内容保护区

```markdown
<!-- MANUAL ADDITIONS START -->
<!-- MANUAL ADDITIONS END -->
```

**说明**：

- 自动生成内容时会保留这两个标记之间的内容
- 用于维护手动添加的配置和说明
- 更新脚本会跳过此区域

## 7. 模版与命令的交互

### 7.1 参数传递

模版通过命令的 `$ARGUMENTS` 占位符接收用户输入。

**命令文件示例**：

```markdown
---
description: 创建或更新功能规范
---

## 用户输入

$ARGUMENTS
```

**模版文件示例**：

```markdown
# 功能规范: [FEATURE NAME]

**输入**: 用户描述: "$ARGUMENTS"
```

### 7.2 数据流

```
用户输入 → 命令($ARGUMENTS) → 模版填充 → 生成文档
```

**示例**：

1. 用户执行命令：`/create-spec 用户认证系统`
2. 命令读取模版文件
3. 替换 `[FEATURE NAME]` → `用户认证系统`
4. 替换 `$ARGUMENTS` → `用户认证系统`
5. 生成最终文档

### 7.3 前置条件检查

命令可以通过脚本检查模版所需的前置条件。

**命令示例**：

```markdown
1. **设置**: 运行前置条件检查脚本
   获取必要的配置信息（如 FEATURE_SPEC, IMPL_PLAN, SPECS_DIR, BRANCH）
```

**脚本功能**：

- 检查必需文件是否存在
- 验证目录结构
- 准备模版变量
- 返回 JSON 格式的配置

## 8. 最佳实践

### 8.1 占位符命名

- 使用大写字母和下划线：`[PROJECT_NAME]`
- 使用描述性名称：`[PRINCIPLE_1_DESCRIPTION]`
- 保持一致性：相同概念使用相同名称

### 8.2 注释使用

- 提供清晰的操作指导
- 说明占位符的预期内容
- 标记重要注意事项
- 提供示例说明

### 8.3 结构设计

- 使用清晰的章节划分
- 保持逻辑层次
- 提供检查点标记
- 支持增量填充

### 8.4 可扩展性

- 使用可变数量的区块（如 `[PRINCIPLE_N_NAME]`）
- 提供可选区块
- 支持条件内容
- 允许手动添加

### 8.5 文档完整性

- 包含所有必需信息
- 提供使用说明
- 说明与命令的关系
- 记录依赖关系

## 9. 模版示例

### 9.1 完整的功能规范示例

```markdown
# 功能规范: 用户认证系统

**功能分支**: `001-user-authentication`
**创建时间**: 2026-03-11
**状态**: 草稿
**输入**: 用户描述: "实现用户注册、登录和权限管理"

## 用户场景与测试 _(必填)_

### 用户故事 1 - 用户注册 (优先级: P1)

作为新用户，我希望能够注册账户，以便使用系统的完整功能。

**优先级原因**: 注册是所有认证功能的基础，没有用户就无法进行后续操作。

**独立测试**: 可以通过注册表单提交、邮箱验证、密码设置流程完全测试，交付可用的注册功能。

**验收场景**:

1. **给定** 用户在注册页面, **当** 填写有效信息并提交, **那么** 创建账户并发送验证邮件
2. **给定** 用户收到验证邮件, **当** 点击验证链接, **那么** 账户激活成功

---

### 用户故事 2 - 用户登录 (优先级: P2)

作为已注册用户，我希望能够登录系统，以便访问我的个人数据。

**优先级原因**: 登录是用户访问系统的主要入口，优先级仅次于注册。

**独立测试**: 可以通过登录表单、会话管理、退出流程完全测试，交付可用的登录功能。

**验收场景**:

1. **给定** 用户在登录页面, **当** 输入正确凭据, **那么** 成功登录并跳转到首页
2. **给定** 用户已登录, **当** 点击退出, **那么** 清除会话并跳转到登录页

---

## 需求 _(必填)_

### 功能需求

- **FR-001**: 系统必须允许用户使用邮箱和密码注册
- **FR-002**: 系统必须验证用户邮箱地址
- **FR-003**: 系统必须支持密码加密存储
- **FR-004**: 系统必须提供登录和退出功能
- **FR-005**: 系统必须实现会话管理

### 关键实体

- **用户**: 系统使用者，包含邮箱、密码哈希、激活状态等属性
- **会话**: 用户登录状态，包含令牌、过期时间等属性

## 成功标准 _(必填)_

### 可衡量的结果

- **SC-001**: 用户可以在 1 分钟内完成注册流程
- **SC-002**: 系统支持 1000 个并发登录请求
- **SC-003**: 99% 的用户在首次尝试时成功登录
```

```markdown
**输入**: 来自 `/specs/001-user-authentication/` 的设计文档
```

## 10. 调试与测试

### 10.1 模版验证

检查模版是否正确：

1. 所有占位符使用正确格式：`[PLACEHOLDER_NAME]`
2. HTML 注释格式正确：`<!-- 注释内容 -->`
3. Markdown 结构完整
4. 必需章节都存在

### 10.2 命令测试

测试命令与模版的交互：

```bash
# 测试功能规范命令
/create-spec 测试功能

# 测试实施计划命令
/create-plan

# 测试任务列表命令
/create-tasks
```

### 10.3 常见问题

| 问题             | 原因           | 解决方案                              |
| ---------------- | -------------- | ------------------------------------- |
| 占位符未被替换   | 占位符格式错误 | 使用 `[PLACEHOLDER]` 格式             |
| 生成文档结构错误 | 模版章节缺失   | 检查必需章节                          |
| 中文乱码         | 编码问题       | 确保 UTF-8 编码                       |
| 手动内容被覆盖   | 未使用保护区   | 添加 `<!-- MANUAL ADDITIONS -->` 标记 |

## 附录：占位符速查表

### A. 通用占位符

| 占位符               | 说明       | 示例值        |
| -------------------- | ---------- | ------------- |
| `[FEATURE]`          | 功能名称   | User Auth     |
| `[FEATURE NAME]`     | 完整功能名 | 用户认证系统  |
| `[DATE]`             | 日期       | 2026-03-11    |
| `[###-feature-name]` | 分支名     | 001-user-auth |
| `[PROJECT_NAME]`     | 项目名     | MyProject     |

### B. 任务相关占位符

| 占位符     | 说明         | 示例值     |
| ---------- | ------------ | ---------- |
| `[T###]`   | 任务编号     | T001, T002 |
| `[US#]`    | 用户故事编号 | US1, US2   |
| `[P]`      | 并行标记     | [P]        |
| `[CHK###]` | 检查项编号   | CHK001     |

### C. 需求相关占位符

| 占位符               | 说明     | 示例值    |
| -------------------- | -------- | --------- |
| `[FR-###]`           | 功能需求 | FR-001    |
| `[SC-###]`           | 成功标准 | SC-001    |
| `[PRINCIPLE_N_NAME]` | 原则名称 | I. 库优先 |
| `[SECTION_N_NAME]`   | 章节名称 | 治理      |

### D. 条件占位符

| 占位符                        | 说明     | 使用场景     |
| ----------------------------- | -------- | ------------ |
| `[NEEDS CLARIFICATION: 描述]` | 需要澄清 | 不明确的需求 |
| `[如未使用请删除]`            | 可选内容 | 项目结构选项 |
| `[可选 - 仅在要求时]`         | 可选章节 | 测试任务     |
