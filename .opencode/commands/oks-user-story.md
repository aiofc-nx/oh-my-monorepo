# 用户故事（User Story）

将功能需求转化为符合 INVEST 原则的用户故事。

```gherkin
作为 <角色>
```

## 执行步骤

### 1. 分析需求

理解 `$ARGUMENTS` 的业务价值：

- 这个功能解决什么问题？
- 目标用户是谁？
- 用户从中获得什么价值？

### 2. 识别角色

确定目标用户类型：

- 主要用户：直接使用功能的人
- 次要用户：间接受益的人
- 系统角色：其他系统或服务

### 3. 定义功能

明确用户想要做什么：

- 核心功能是什么？
- 需要哪些输入？
- 产生哪些输出？

### 4. 明确价值

说明用户从中获得的收益：

- 提高效率？
- 降低成本？
- 改善体验？
- 降低风险？

### 5. 设定优先级

为每个用户故事分配优先级：

| 优先级 | 说明               | 示例     |
| ------ | ------------------ | -------- |
| **P1** | 核心功能，必须实现 | 用户登录 |
| **P2** | 重要功能，应该实现 | 记住我   |
| **P3** | 增强功能，可以实现 | 登录日志 |

---

## INVEST 原则检查

| 原则            | 说明   | 检查点               | 通过标准          |
| --------------- | ------ | -------------------- | ----------------- |
| **I**ndependent | 独立性 | 故事之间没有依赖关系 | ✅ 可独立开发     |
| **N**egotiable  | 可协商 | 细节可以讨论         | ✅ 技术方案灵活   |
| **V**aluable    | 有价值 | 对用户有明确价值     | ✅ 用户价值明确   |
| **E**stimable   | 可估算 | 能够估算工作量       | ✅ 工作量 < 3 天  |
| **S**mall       | 足够小 | 一个迭代内能完成     | ✅ 单次提交可完成 |
| **T**estable    | 可测试 | 有明确的验收标准     | ✅ 验收标准明确   |

## 常见问题

### Q: 如果功能太大怎么办？

A: 拆分成多个小故事，每个分配优先级：

```
大故事: 用户管理
  ├─ P1: 用户登录
  ├─ P1: 用户注册
  ├─ P2: 用户资料编辑
  └─ P3: 用户登出
```

### Q: 如果验收标准不明确怎么办？

A: 使用 Given-When-Then 格式细化：

```
给定 [初始状态]
当 [执行操作]
那么 [期望结果]
```

### Q: 如果故事之间有依赖怎么办？

A: 重新组织或合并：

- 合并强依赖的故事
- 调整优先级顺序
- 识别并提取公共部分

### Q: 如何标记待澄清项？

A: 使用 `[NEEDS CLARIFICATION: 具体问题]` 格式：

```markdown
- **FR-005**: 系统必须支持多语言 - [NEEDS CLARIFICATION: 未指定支持哪些语言]
```

### Q: 如何设定成功标准？

A: 成功标准必须是可衡量的：

```markdown
- **SC-001**: 用户可以在 X 分钟内完成 [操作] ✅ 好的标准
- **SC-002**: 系统性能良好 ❌ 不可衡量
- **SC-003**: API 响应时间 < 200ms ✅ 好的标准
```
